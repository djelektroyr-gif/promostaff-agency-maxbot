# Дорожная карта интеграции с T-API (Т-Банк)

**Дата:** 2026-04-17  
**Источник нормативов API:** [T-API — документация](https://developer.tbank.ru/docs/api), [Начало работы](https://developer.tbank.ru/docs/intro), [Бизнес-сценарии](https://developer.tbank.ru/docs/products).  
**Аудитория:** визитки (Telegram, MAX), веб-панель, бэкенд учёта смен и выплат.

---

## 1. Что даёт T-API в целом

По [введению в T-API](https://developer.tbank.ru/docs/intro): доступ для **юрлиц и ИП**, работа через запросы/ответы, **токены**, **сертификаты**, **вебхуки**, **песочница** для тестов. Это позволяет:

- уменьшить ручной ввод платежей и сверок в интернет-банке;
- получать **статусы** операций в своей системе (веб-панель, уведомления в ботах);
- строить **единый контур** «заказ → договор/закрывающие → оплата заказчика → выплата исполнителю» с журналированием в БД.

Интеграция **не заменяет** сбор анкет, ПДн и верификацию по продуктовой спецификации — она **надстраивается** поверх уже собранных данных (`REGISTRATION_AND_POST_MENU_SPEC.md`).

## 2. Разделы T-API и применимость к PROMOSTAFF

Ниже — группы из [каталога API](https://developer.tbank.ru/docs/api) с краткой пометкой, нужны ли они нам.

| Раздел T-API | Нужность для нас | Зачем |
|--------------|------------------|--------|
| **T-ID** (в т.ч. партнёрская интеграция) | **Высокая** | Вход в веб-панель «как в Т-Бизнес», привязка пользователя к юрлицу/ИП без дублирования паспорта там, где банк выступает идентификатором. Сценарии: [продукты → T-ID](https://developer.tbank.ru/docs/products). |
| **Счета и выписки** | **Высокая** | Автоматическая **сверка поступлений** от заказчиков, выгрузка выписок за период, сопоставление с заявками/счетами в CRM. |
| **Платежи** | **Средняя–высокая** | Исходящие переводы по реквизитам (где нет отдельного продукта «самозанятые»), контроль статусов. |
| **Зарплатный проект** | **Низкая–средняя** | Имеет смысл, если появится **много штатных** сотрудников; для основной массы **самозанятых** приоритетнее блок ниже. |
| **Выплаты самозанятым** | **Очень высокая** | Прямое попадание в модель: исполнители с НПД, ИНН и чеками; массовые/поэтапные выплаты после закрытия смен из веб-панели. Сценарий на [странице продуктов](https://developer.tbank.ru/docs/products) («Работа с самозанятыми»). |
| **Выставление ссылок через СБП** | **Высокая** | Быстрая оплата от заказчика по ссылке из бота/письма/ЛК; меньше трения, чем «вышлите реквизиты вручную». |
| **Выставление счетов** | **Высокая** | Юрлицо–юрлицо: счёт на услуги агентства, привязка к проекту в панели. |
| **Номинальные счета** | **Средняя (по бизнес-модели)** | Если понадобится **раздельный учёт средств** по проектам/заказчикам (агентская схема) — уточнять с финансами и юристами. |
| **Безакцептные списания** | **Низкая** | Только при заранее согласованных договорах и подписках; не базовый сценарий визитки. |
| **Торговый эквайринг** | **Опционально** | Если продажи «картой на сайте» не через СБП-ссылки. |
| **Бизнес-карты** | **Низкая** | Операционные расходы, не ядро продукта. |
| **Партнёрские интеграции** | **По мере масштаба** | Если сервис выступает **платформой для многих клиентов** — отдельный трек подключения ([партнёрская интеграция](https://developer.tbank.ru/docs/intro/partner/)). |
| **Работа с файлами / Письма** | **Средняя** | Вложения, акты, сканы — если банк или процесс обмена документами идёт через T-API. |
| **ВЭД, гарантии, ESOP и др.** | **Не в фокусе** | На старте не планируем. |

**Инфраструктура (обязательно учесть в плане):**

- Подключение: [прямая H2H](https://developer.tbank.ru/docs/intro/manuals/) или [партнёрская](https://developer.tbank.ru/docs/intro/partner/).
- [Токены](https://developer.tbank.ru/docs/intro/manuals/self-service-auth), [сертификаты](https://developer.tbank.ru/docs/intro/manuals/certificates), [вебхуки](https://developer.tbank.ru/docs/intro/webhooks/), [песочница](https://developer.tbank.ru/docs/intro/other/sandbox).

## 3. Какие «функции» (в продуктовом смысле) нам понадобятся

Сгруппировано по контурам, а не по точным именам методов OpenAPI (их перечень — в личном кабинете и в спецификации после подключения продукта).

### 3.1. Веб-панель (приоритет)

1. **Аутентификация и привязка организации** — T-ID / сценарий из раздела T-ID, чтобы заказчик или админ агентства не плодил расхождений между «наш ИНН в анкете» и «счёт в банке».
2. **Выписки и остатки** — загрузка движений по счёту, сопоставление с сущностями «проект / счёт / оплата заказчика».
3. **Выставление счетов** — направление счетов контрагентам (заказчикам) из карточки проекта.
4. **СБП-ссылки** — генерация ссылки на оплату и отображение статуса в панели (и при желании — копирование в Telegram-менеджеру).
5. **Выплаты самозанятым** — из реестра смен: ИНН/идентификатор получателя, сумма, основание; обработка статусов и ошибок (неверный ИНН, лимиты НПД и т.д.).
6. **Вебхуки** — обновление статусов платежей и выплат без опроса «в лоб».

### 3.2. Боты-визитки (Telegram / MAX)

- **Короткий контур:** уведомление «счёт выставлен» / «ссылка на оплату готова» / «оплата получена» — через **веб-панель + бэкенд**, бот только доставляет сообщение (без хранения банковских секретов в боте).
- **Не целимся** в полный банковский функционал внутри MAX/Telegram: секреты, подпись запросов и сертификаты остаются на **сервере агентства**.

### 3.3. MAX — верификация исполнителей (заложить в план, не делать сейчас)

- **Сейчас:** полный админ-контур верификации в MAX **не обязателен**; канон — **Telegram + веб-панель** (как согласовано в roadmap продукта).
- **Позже:** довести уведомления/действия админа для заявок из MAX до паритета с Telegram (те же статусы в БД, те же кнопки «верифицировать / отказ» в интерфейсе, который выберете — веб или отдельный канал).
- **Связь с T-Банком:** выплаты идут по данным из БД после верификации **независимо от канала** (MAX/Telegram); важно единое `executor_id` и статус «допущен к выплатам».

## 4. Фазы внедрения (рекомендация)

| Фаза | Содержание | Результат для процессов |
|------|------------|-------------------------|
| **0. Подготовка** | Договор с банком, выбор сценария (H2H vs партнёрский), песочница, токен, сертификат, тестовые вебхуки | Безопасный контур разработки |
| **1. Наблюдение** | Выписки + ручная сверка с проектами в панели | Меньше ручного захода в Т-Бизнес |
| **2. Дебиторка** | Счета + СБП-ссылки из веб-панели, статусы в карточке заказчика | Быстрее оплата, меньше «потерянных» счетов |
| **3. Кредиторка (самозанятые)** | Выплаты самозанятым из закрытых смен, вебхуки статусов | Масштабирование выплат без операционки «по файлам» |
| **4. Углубление** | Номинальные счета / зарплатный проект — по решению финмодели | Отдельный юридический и продуктовый трек |

## 5. Упрощение разработки и автоматизация

- **Меньше дублирования:** статусы денег живут в одной системе (панель + БД), а не «в Excel + в банке».
- **Меньше ошибок:** ИНН и получатель берутся из уже проверенной анкеты; ошибки банка возвращаются в UI.
- **Прозрачность для исполнителя:** в пункте «Мои выплаты» (меню после регистрации) можно показывать **статусы из T-API** вместо статичной заглушки.
- **Прозрачность для заказчика:** «История заказов» + привязка к оплате (счёт/СБП) без звонков менеджеру.
