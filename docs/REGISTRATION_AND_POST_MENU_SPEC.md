# Регистрация и меню после регистрации — каноническая спецификация

**Версия:** 1.3  
**Статус:** единственный источник правды для `promostaff-agency-bot` и `promostaff-agency-maxbot`.  
**Дублирование:** идентичная копия лежит в `promostaff-agency-bot/docs/REGISTRATION_AND_POST_MENU_SPEC.md` — любые изменения вносятся **в оба файла одновременно**.

---

## 1. Назначение документа

Зафиксировать:

- полный объём **регистрации** (без сокращения шагов и блоков относительно утверждённой модели Pro);
- единое определение **«конец регистрации»**;
- состав **главного меню** по ролям после регистрации;
- обязательные отличия платформ (Telegram / MAX), не урезающие смысл и полноту данных.

Отклонения от этого документа в коде считаются **дефектом**, пока документ не изменён осознанно и под версией.

Актуальная **сверка кода с ТЗ Pro и этим документом** ведётся в `docs/TZ_GAP_ANALYSIS.md` (дублируется в `promostaff-agency-bot/docs/`).

---

## 2. Нормативная база по полноте регистрации

Полнота анкет, согласий и проверок для исполнителя и заказчика приводится в соответствие с **«ФИНАЛЬНОЕ ТЗ PROMOSTAFF PRO — Бот для управления проектами в MAX» (v11.x и далее)** — разделы про регистрацию заказчика и исполнителя, каталог профессий, верификацию, правовые формулировки.

**Упрощать блок регистрации запрещено:** нельзя удалять шаги, объединять юридически значимые согласия в один клик без отдельного решения, убирать сбор обязательных полей или заменять их «заглушками», если они присутствуют в нормативном ТЗ для данной роли.

Интеграции (веб-панель, T-Банк, ЕСИА/Connect ID) подключаются **поверх** этой модели данных, а не вместо неё. Детализация по T-API (счета, выписки, СБП, выплаты самозанятым, T-ID): `docs/TBANK_T_API_ROADMAP.md` (дубликат в `promostaff-agency-bot/docs/`).

---

## 3. «Конец регистрации» — определение

**Регистрация считается завершённой** в момент, когда пользователь после прохождения всех обязательных шагов своей роли впервые получает **главное меню этой роли** (см. §5–6).

До этого момента пользователь находится в сценарии регистрации (FSM / `SESSIONS` и т.п.).

**Неверно** считать концом регистрации автоматический переход заказчика в оформление заказа/выбор режима расчёта без показа меню заказчика — такое поведение не соответствует данной спецификации.

## 4. Регистрация: обязательные элементы (без сокращения)

### 4.1. Заказчик (юрлицо / ИП)

Сохраняется полная логика из нормативного ТЗ Pro, включая:

- ввод и проверка **ИНН**;
- подтверждение данных организации (**ЕГРЮЛ/ЕГРИП**);
- **верификация представителя** через предусмотренный канал (**Connect ID / Госуслуги**), когда доступен в MAX; при недоступности канала — зафиксированный в релизе **план Б** (ручная/админская верификация) без удаления шагов сбора реквизитов и согласий;
- все необходимые **согласия на обработку персональных данных** и иные юридически значимые акцепты в порядке, заданном ТЗ.

Данные организации и контактов должны быть сохранены в системе учёта до выдачи меню заказчика.

### 4.2. Исполнитель

Сохраняется **полная** регистрация по нормативному ТЗ Pro, включая (без исчерпывающего сокращения перечня из ТЗ):

- ФИО, телефон, дата рождения;
- налоговый статус и ветки (физлицо / самозанятый / ИП) с запросом подтверждающих данных, где требуется ТЗ;
- опыт, описание, выбор профессии из каталога (и сценарий «новая профессия» по ТЗ);
- параметры (рост, вес, размеры), форма, документы (медкнижка и т.д.), навыки;
- **селфи** и путь верификации (AI/админ) по ТЗ;
- промежуточные и финальные согласия в порядке, заданном ТЗ.

### 4.3. Платформа MAX — дополнительный обязательный шаг

**Перед шагом загрузки селфи** пользователю показывается экран с:

- ссылками на **Политику конфиденциальности** и **Пользовательское соглашение** (`https://promostaff.pro/privacy`, `https://promostaff.pro/terms` или актуальные URL из продукта);
- кнопками **«Согласен»** / **«Не согласен»** (или эквивалент в API MAX).

При отказе регистрация **не завершается**; сценарий выхода фиксируется в реализации (например, возврат к пояснению или в стартовое меню без роли).

Этот шаг **не заменяет** иные согласия (в т.ч. на ПДн), а дополняет их перед финализацией медиаданных.

### 4.4. Паритет Telegram и MAX

Содержательно шаги регистрации **одинаковы по полноте**; отличаются только механизмы UI (inline/callback, вложения, ограничения API). Урезание полей на одной платформе относительно другой **не допускается**, если не оформлено изменением данного документа.

### 4.5 Какой сценарий считать правильным (исполнитель) — статус в коде

Нормативная модель по полноте данных — по-прежнему §4.2 и ТЗ Pro. **Эталон реализации по шагам и полям анкеты визитки агентства** (новый согласованный сценарий: профессия из каталога, ФИО/телефон/дата рождения, налоговый статус и ветки со справками, опыт, параметры, форма, медкнижка, навыки, согласия, селфи, паспорт, проверка данных, отправка и **верификация админом до меню исполнителя**):

| Платформа | Где в коде | Заметки |
|-----------|------------|---------|
| **Telegram** | `join_anketa_handlers.py` + состояния `JoinForm` в `states.py`; вход из `handlers.py` (`join_team`, `fill_anketa`, отклик по вакансии → согласие ПДн → `join_proceed_anketa`). | **Один активный сценарий.** Старые блоки «трек / docs_med / смены из одной клавиатуры» в том виде, как в ранней визитке, **в маршрутизацию не подключены** (см. неиспользуемые функции в `keyboards.py`: `join_profile_keyboard`, `join_shift_pref_keyboard`, `join_docs_keyboard`, `submit_join_keyboard` и связанные callback — мёртвый остаток, не дублируют активный FSM). |
| **MAX** | `visit_flows.py` + `visit_card.py`. | **Частичный паритет с Telegram:** добавлены дата рождения и налоговый блок (`tax_*`), синхронизированные по смыслу с новой анкетой. Ниже по воронке пока остаются **упрощённые шаги** прошлой анкеты (выбор «трека» `jp_*`, блок **`docs_ready`** и др.) — это **не** полное совпадение с `join_anketa_handlers`. Имеется **смысловое пересечение**: статус СЗ/ИП уже фиксируется в налоговом шаге, а `docs_ready` дополнительно спрашивает про самозанятость/ИП; дальнейшая унификация — **точечными правками** (сузить `docs_ready` до медкнижки/«позже» или убрать дублирование). |

**Итог для продукта:** правильной (целевой) регистрацией исполнителя визитки по содержанию анкеты является сценарий **`join_anketa_handlers.py` (Telegram)**; MAX нужно постепенно подтягивать к тому же объёму и порядку, без параллельного «второго полного» сценария в Telegram.

**После отправки анкеты:** меню исполнителя выдаётся только после **верификации администратором** (кнопки/веб-панель, HRM, `verification_status` в payload) — см. реализацию в `db.py` / панели агентства.

## 5. Меню заказчика после регистрации

После завершения регистрации заказчик получает **главное меню заказчика** со следующими пунктами (первый этап — допускаются заглушки там, где указано):

| Пункт | Назначение |
|--------|------------|
| **Мои проекты** | Список и статусы проектов пользователя/организации, связь с учётом в promostaff-bot / панелью по мере готовности данных. |
| **История заказов** | Хронология заявок (визитка, КП, срочные и т.д.), привязка к CRM/`agency_visit_orders` и связанным сущностям. |
| **Настройки** | Заглушка до появления настроек (уведомления, профиль и т.д.). |
| **Открыть веб-панель** | Заглушка/ссылка на вход в веб-панель агентства; финальный URL и SSO — отдельно (в т.ч. T-Банк). |

Оформление **нового** заказа / расчёта выполняется **отдельным действием** из этого меню или из связанных экранов, а не автоматически сразу после регистрации.

---

## 6. Меню исполнителя после регистрации

После завершения регистрации исполнитель получает **главное меню исполнителя**:

| Пункт | Назначение |
|--------|------------|
| **Мои данные** | Просмотр и редактирование анкеты в рамках правил верификации (изменения, требующие повторной проверки — по политике из ТЗ). |
| **Мои смены** | Прошлые, текущие, будущие смены при наличии назначений в данных. |
| **Мои выплаты** | Список и статусы выплат; до готовности интеграций — заглушка или минимальный список из доступной БД. |
| **Маяк** | Включение/выключение режима готовности к **срочному поиску** (логика окна активности, автоотключение — по ТЗ Pro, например 6 часов). |

---

## 7. Связь с promostaff-bot и внешними системами

- Учёт проектов, смен, маяка/маяка в БД, панель агентства — **наращиваются** в соответствии с этим документом и ТЗ Pro, без противоречия §2.
- **T-Банк** и прочие платёжные/идентификационные интеграции подключаются на этапе реализации; они **не являются основанием** для сокращения регистрации.
