# Сверка с ТЗ Pro и `REGISTRATION_AND_POST_MENU_SPEC.md`

**Дата:** 2026-04-17  
**Назначение:** зафиксировать расхождения на момент проверки и план приведения кода к канону.

Нормативные документы:

- `docs/REGISTRATION_AND_POST_MENU_SPEC.md` (v1.0) — продуктовая спецификация agency-ботов.
- «ФИНАЛЬНОЕ ТЗ PROMOSTAFF PRO…» (v11.x) — полнота анкет, Connect ID, каталог профессий, маяк и т.д.

---

## 1. Меню после регистрации (§3, §5–6 спецификации)

| Требование | Было (до правки) | Целевое состояние |
|------------|------------------|-------------------|
| Конец регистрации заказчика | Переход в `OrderForm.order_mode` (Telegram и MAX) | Главное меню заказчика |
| Конец регистрации исполнителя | Главное меню визитки (`main_menu_keyboard`) | Главное меню исполнителя |
| Пункты меню заказчика | Не реализованы как единое меню | Мои проекты, История заказов, Настройки (заглушка), Открыть веб-панель (заглушка) + отдельно «Заказать расчёт» |

**Статус кода:** после подтверждения данных заказчика показывается **меню заказчика** (`client_registered_main_menu_keyboard`), без автоперехода в выбор типа расчёта (Telegram: `registration_flow.py`; MAX: `visit_flows.py`). Пункты «Мои проекты», «Настройки», «Веб-панель» — заглушки; «История заказов» — список заявок; новый расчёт — через «Заказать расчёт».

---

## 2. Регистрация заказчика vs ТЗ Pro (§4.1)

| Элемент ТЗ Pro | Текущая реализация agency-ботов |
|----------------|----------------------------------|
| ИНН + проверка | Есть ввод ИНН по длине; **нет** запроса к ЕГРЮЛ/ЕГРИП и экрана «Найдено в реестре» |
| Connect ID / Госуслуги | **Нет** |
| План Б (ручная верификация) | Частично: данные сохраняются, админ видит заявки; **не** оформлено как отдельный сценарий ТЗ |
| Полный набор согласий | Есть согласие ПДн на визитку; **нет** полного паритета с v11 по всем акцептам |

**Вывод:** блок «визитка заказчика» **короче**, чем регистрация заказчика в ТЗ Pro. Дальнейшая работа — наращивание шагов (ЕГРЮЛ, Connect ID при доступности API MAX, отдельные экраны) **без удаления** уже собираемых полей.

---

## 3. Регистрация исполнителя vs ТЗ Pro (§4.2)

| Элемент ТЗ Pro (примеры) | Текущая реализация `JoinForm` / MAX `join` |
|--------------------------|--------------------------------------------|
| Дата рождения | **Нет** |
| Налоговый статус, ветка самозанятого, ИНН/скрин | **Нет** |
| Категории профессий + 38 профессий | Упрощённо: треки + список `APPLICANT_POSITIONS` |
| Рост, вес, размеры одежды/обуви | **Нет** |
| Отдельный блок «форма одежды» по ТЗ | **Нет** |
| Навыки чекбоксами + текст | Только текст |
| Селфи + верификация AI/админ | Селфи есть; **нет** полного контура верификации Pro |

**Вывод:** анкета agency — **укороченная** относительно v11. Согласно канону, её нужно **расширять** до полноты ТЗ, а не заменять «упрощённой» версией навсегда.

---

## 4. Шаг privacy/terms перед селфи (§4.3 спецификации)

| Платформа | Было | После правки |
|-----------|------|----------------|
| MAX | Сразу запрос селфи после навыков | Экран ссылок + «Согласен»/«Не согласен» **перед** селфи |
| Telegram | Сразу селфи | То же: состояние `JoinForm.terms_accept`, кнопки и ссылки на privacy/terms **перед** селфи |

---

## 5. Паритет Telegram / MAX (§4.4)

- Сценарии заказа и визитки **содержательно** совпадают по шагам в разумных пределах API.
- Полная идентичность полей исполнителя **пока не достигнута** — см. §3.

---

## 6. Роль после `/start` и повторных входов

- После внедрения ролевых меню сразу после регистрации пользователь видит корректное меню **один раз** в конце сценария.
- **Не сделано:** автоматическое определение роли при каждом `/start` по БД (заказчик всегда видит меню заказчика). Это следующий этап: флаги в БД или связь `user_id` с ролью.

---

## 7. Приоритеты работ (рекомендация)

1. **Сделано:** ролевые меню после регистрации; Telegram и MAX — шаг terms/privacy перед селфи в анкете исполнителя.
2. **Далее:** расширение анкеты исполнителя и регистрации заказчика до полноты ТЗ Pro (итерациями).
3. **Далее:** Connect ID / ЕГРЮЛ при готовности API и юридики.
4. **Далее:** сохранение роли и показ ролевого меню при `/start`.
5. **Интеграции:** веб-панель и **T-API (Т-Банк)** — см. детальный план в `docs/TBANK_T_API_ROADMAP.md` (фазы: выписки → счета/СБП → выплаты самозанятым; боты получают только уведомления с бэкенда). Интеграции **не сокращают** регистрацию.
6. **MAX — админ-верификация исполнителей:** на текущем этапе **не** доводим до полного паритета с Telegram; целевой паритет и те же статусы в БД — **заложено в бэклог** (после стабилизации TG + веб-панели). На выплаты через T-API это не влияет, если в БД единый профиль исполнителя и флаг «допущен к выплатам».
